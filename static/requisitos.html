Requisitos do Sistema — CondConnect

Quadro 4 – Requisitos Funcionais

IDTítuloDescrição
RF01 Cadastro de usuários O sistema deve permitir o cadastro de usuários informando nome, e-mail único e senha. O papel padrão atribuído é membro. O usuário pode complementar o perfil com foto e chave Pix após o cadastro.
RF02 Autenticação / Login O sistema deve permitir o login de usuários cadastrados mediante autenticação com e-mail e senha. Deve oferecer fluxo de recuperação de senha por e-mail com token temporário.
RF03 Cadastro de produto O sistema deve permitir que o usuário cadastre produtos que deseja anunciar, com campos para título, descrição, categoria, preço, condição (Novo / Seminovo / Usado) e upload de múltiplas imagens. O produto é publicado com status ativo.
RF04 Busca de produtos O sistema deve disponibilizar funcionalidade de busca e filtragem de produtos, permitindo localizar anúncios por categoria, condição e faixa de preço. Cada resultado deve exibir o rating médio calculado via subquery SQL.
RF05 Edição e exclusão de anúncio O sistema deve permitir a edição e exclusão de anúncios pelo usuário que os criou. A remoção também pode ser realizada pelo administrador em casos de violação das regras da plataforma.
RF06 Visualização de anúncio O sistema deve permitir que o usuário visualize os detalhes de um anúncio, incluindo título, descrição, preço, condição, imagens, rating médio, total de avaliações e chave Pix do vendedor.
RF07 Carrinho e pedido O sistema deve permitir que o comprador adicione itens ao carrinho (tabela itens_carrinho) e finalize a compra gerando um pedido. O vendedor atualiza o status do pedido (pendente → enviado → entregue ou cancelado). O comprador não pode comprar produtos anunciados por si mesmo.
RF08 Avaliação de produto O sistema deve permitir que o comprador avalie o produto adquirido com nota de 1 a 5 e comentário opcional, após a confirmação de entrega. A avaliação é vinculada ao produto, não ao vendedor diretamente.
RF09 Chave Pix do vendedor O sistema deve permitir que o vendedor cadastre e exiba sua chave Pix no perfil, para que o comprador realize o pagamento externamente. A plataforma não integra gateway de pagamento.
RF10 Mensageria interna Requisito removido do escopo. A funcionalidade de chat entre comprador e vendedor não foi implementada na versão atual do sistema. A negociação ocorre por meio do fluxo de propostas (RF14).
RF11 Administração e moderação O sistema deve permitir que o administrador gerencie usuários (bloquear contas) e produtos (remover anúncios inapropriados). O acesso ao painel é restrito a usuários com papel admin, verificado em cada requisição.
RF12 Anúncios de empresas externas Requisito removido do escopo. A funcionalidade de espaço publicitário para empresas externas ao condomínio foi descartada para manter o foco do sistema no comércio entre moradores.
RF13 Favoritar produto O sistema deve permitir que o usuário adicione e remova produtos da sua lista de favoritos. Os favoritos são persistidos na tabela favoritos e sincronizados com o localStorage no frontend.
RF14 Negociação por proposta O sistema deve permitir que o comprador envie proposta de negociação contendo quantidade e valor ao vendedor. O vendedor aceita ou recusa. Ao aceitar, o preço negociado é registrado no campo preco_negociado da tabela itens_carrinho.
RF15 Painel administrativo O sistema deve disponibilizar painel administrativo restrito a usuários com papel admin, com gestão de usuários, moderação de produtos, visualização de pedidos e indicadores agregados (total de usuários, produtos ativos, pedidos e receita total) calculados via SQL.
RF16 Upload de imagens O sistema deve permitir o upload de múltiplas imagens por produto via multipart/form-data, com geração de nome único por UUID e armazenamento no servidor. Uma das imagens deve ser marcada como principal para exibição em listagens.
RF17 Notificações internas O sistema deve emitir notificações internas aos usuários para eventos como: proposta recebida, proposta aceita ou recusada e atualização de status de pedido. Deve exibir badge com contador de notificações não lidas e permitir marcação em lote como lidas.
RF18 Atualização de status do pedido O sistema deve permitir que o vendedor atualize o status do pedido (pendente → enviado → entregue ou cancelado). Após o status entregue, o comprador fica habilitado a avaliar o produto.

Fonte: Os Autores (2025).

RF10 (Mensageria interna) e RF12 (Anúncios de empresas) foram mantidos na tabela com a identificação de removidos para preservar a rastreabilidade com o histórico de alterações (Quadro 5).

Quadro 6 – Requisitos Não Funcionais

IDTítuloDescrição
RNF01 Responsividade O sistema deve possuir interface responsiva com abordagem mobile-first, adaptando-se a diferentes tamanhos de tela e dispositivos (computadores, tablets e smartphones), sem dependência de frameworks CSS externos.
RNF02 Segurança e criptografia O sistema deve garantir autenticação segura via sessão server-side (Flask-Login com cookie HttpOnly) e armazenar senhas exclusivamente como hash criptográfico (Werkzeug). Dados sensíveis não devem ser expostos em respostas da API.
RNF03 Desempenho (tempo de resposta) O tempo de resposta para operações comuns (login, busca, carregamento de anúncios) não deve exceder 3 segundos em condições normais de uso.
RNF04 Concorrência / Escalabilidade O sistema deve suportar acesso simultâneo de múltiplos usuários sem degradação perceptível de desempenho, por meio de servidor WSGI (Gunicorn) com múltiplos workers.
RNF05 Disponibilidade O sistema deve apresentar disponibilidade mínima de 95%, considerando janelas de manutenção programada comunicadas com antecedência aos usuários.
RNF06 Usabilidade A interface deve ser intuitiva e de fácil navegação, priorizando clareza visual, hierarquia de informação e consistência de padrões de interação, sem necessidade de treinamento prévio para uso básico.
RNF07 Rastreabilidade de sessões O sistema deve registrar logs de acesso em nível de servidor (Apache / Gunicorn), possibilitando auditoria de requisições. O rastreamento de ações críticas em nível de aplicação (banco de dados) está previsto para versões futuras do sistema.
RNF08 Qualidade de código / Versionamento O código deve ser desenvolvido seguindo boas práticas de programação (nomenclatura clara, separação de responsabilidades) e versionado em repositório Git, com histórico de commits rastreável.
RNF09 Conformidade LGPD O sistema deve observar a Lei Geral de Proteção de Dados (LGPD — Lei nº 13.709/2018), garantindo que dados pessoais (nome, e-mail, foto, chave Pix) sejam usados apenas para o funcionamento da plataforma, com senhas nunca armazenadas em texto claro.
RNF10 Arquitetura em camadas O sistema deve adotar separação clara entre camada de apresentação (HTML5 / CSS3 / JavaScript ES6+), camada de lógica de negócio (Python 3 / Flask) e camada de persistência (MySQL), comunicando-se via API REST sobre HTTP/HTTPS com payloads em JSON.
RNF11 Stack tecnológica definida O back-end deve ser desenvolvido em Python 3 com o microframework Flask, servido por Gunicorn (WSGI) com Apache HTTP Server como proxy reverso. O front-end deve ser construído em HTML5, CSS3 e JavaScript ES6+ sem frameworks externos. O banco de dados MySQL deve ser acessado via PyMySQL com DictCursor, sem ORM.

Fonte: Os Autores (2025).
